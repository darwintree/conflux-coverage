//! This module provides algorithms for accessing and updating a Merkle
11
//! Accumulator structure persisted in a key-value store. Note that this doesn't
12
//! write to the storage directly, rather, it reads from it via the `HashReader`
13
//! trait and yields writes via an in memory `HashMap`.
14
//!
15
//! # Merkle Accumulator
16
//! Given an ever growing (append only) series of "leaf" hashes, we construct an
17
//! evolving Merkle Tree for which proofs of inclusion/exclusion of a leaf hash
18
//! at a leaf index in a snapshot of the tree (represented by root hash) can be
19
//! given.
20
//!
21
//! # Leaf Nodes
22
//! Leaf nodes carry hash values to be stored and proved. They are only appended
23
//! to the tree but never deleted or updated.
24
//!
25
//! # Internal Nodes
26
//! A non-leaf node carries the hash value derived from both its left and right
27
//! children.
28
//!
29
//! # Placeholder Nodes
30
//! To make sure each Leaf node has a Merkle Proof towards the root, placeholder
31
//! nodes are added so that along the route from a leaf to the root, each node
32
//! has a sibling. Placeholder nodes have the hash value
33
//! `ACCUMULATOR_PLACEHOLDER_HASH`
34
//!
35
//! A placeholder node can appear as either a Leaf node or a non-Leaf node, but
36
//! there is at most one placeholder leaf at any time.
37
//!
38
//! # Frozen Nodes & Non-frozen Nodes
39
//! As leaves are added to the tree, placeholder nodes get replaced by
40
//! non-placeholder nodes, and when a node has all its descendants being
41
//! non-placeholder, it becomes "Frozen" -- its hash value won't change again in
42
//! the event of new leaves being added. All leaves appended (not counting the
43
//! one possible placeholder leaf) are by definition Frozen.
44
//!
45
//! Other nodes, which have one or more placeholder descendants are Non-Frozen.
46
//! As new elements are appended to the accumulator the hash value of these
47
//! nodes will change.
48
//!
49
//! # Leaf Count
50
//! Given a count of the number of leaves in a Merkle Accumulator it is possible
51
//! to determine the shape of the accumulator -- which nodes are filled and
52
//! which nodes are placeholder nodes.
53
//!
54
//! Example:
55
//! Logical view of a Merkle Accumulator with 5 leaves:
56
//! ```text
57
//!            Non-fzn
58
//!           /       \
59
//!          /         \
60
//!         /           \
61
//!       Fzn2         Non-fzn
62
//!      /   \           /   \
63
//!     /     \         /     \
64
//!    Fzn1    Fzn3  Non-fzn  [Placeholder]
65
//!   /  \    /  \    /    \
66
//!  L0  L1  L2  L3 L4   [Placeholder]
67
//! ```
68
//!
69
//! # Position and Physical Representation
70
//! As a Merkle Accumulator tree expands to the right and upwards, we number
71
//! newly frozen nodes monotonically. One way to do it is simply to use in-order
72
//! index of nodes, and this is what we do for the in-memory representation. We
73
//! call the stated numbers identifying nodes below simply "Position", and
74
//! unless otherwise stated, this is the in-order position.
75
//!
76
//! For writing to disk however, we write all the children of a node before the
77
//! parent. Thus for disk write order, it is more convenient to use the
78
//! post-order position as an index. And with that we can map a Merkle
79
//! Accumulator into a key-value storage: key is the post-order position of a
80
//! node, and the value is hash value it carries.
81
//!
82
//! We store only Frozen nodes, and generate non-Frozen nodes on the fly when
83
//! accessing the tree. This way, the physical representation of the tree is
84
//! append-only, i.e. once written to physical storage, nodes won't be either
85
//! modified or deleted.
86
//!
87
//! Here is what we persist for the logical tree in the above example:
88
//!
89
//! ```text
90
//!          Fzn2(6)
91
//!         /      \
92
//!        /        \
93
//!    Fzn1(2)       Fzn3(5)
94
//!   /     \       /     \
95
//!  L0(0)  L1(1)  L2(3)  L3(4)  L4(7)
96
//! ```
97
//!
98
//! When the next leaf node is persisted, the physical representation will be:
99
//!
100
//! ```text
101
//!          Fzn2(6)
102
//!         /      \
103
//!        /        \
104
//!    Fzn1(2)       Fzn3(5)       Fzn4(9)
105
//!   /     \       /     \       /      \
106
//!  L0(0)  L1(1)  L2(3)  L3(4)  L4(7)   L5(8)
107
//! ```
108
//!
109
//! The numbering corresponds to the post-order traversal of the tree.
110
//!
111
//! To think in key-value pairs:
112
//! ```text
113
//! |<-key->|<--value-->|
114
//! |   0   | hash_L0   |
115
//! |   1   | hash_L1   |
116
//! |   2   | hash_Fzn1 |
117
//! |  ...  |   ...     |
118
//! ```
